# Pitfalls: 知识图谱/个人 Wiki 项目常见致命错误

> 针对「钢笔乐高式原子标签知识图谱」项目，按风险等级排列。

---

## 1. 内容冷启动陷阱（Cold-Start Trap）

**问题**：图谱架构再完美，没有足量内容就是空壳。漫游体验依赖链接密度——节点少时每个词条都是死胡同，用户一两次就失去兴趣。

**警示信号**：
- 花了两周搭系统，词条不到 50 个
- 词条平均链接数 < 3（无法形成漫游路径）
- 投入时间中"工具/架构" > "内容"

**预防策略**：
- **先批量导入再打磨体验** — 第一个里程碑的目标是 300+ 原子节点 + 50+ 可浏览实体页面，而非完美 UI
- 利用已有素材库（richardspens 14 目录、品牌款型库）做自动化初始导入
- 设置硬指标：系统能用之前，内容覆盖率必须 > 阈值
- 人工标注 30-50 条"黄金样本"作为 AI 标注的 few-shot 基准

**对应阶段**：Phase 1（数据建模 + 批量导入 pipeline）

## 2. 标签颗粒度失控（Tag Granularity Explosion）

**问题**：原子标签"宁细不粗"是对的，但没有约束就会爆炸——同一概念多种表达（`14K金尖` vs `14k 金笔尖` vs `14ct nib`）、层级含混（`日本制造` 和 `百乐` 同时存在但无层级关系）、数量膨胀到无法维护。

**警示信号**：
- 标签总数 > 实体数 × 10 且持续增长
- 搜索/筛选时出现大量"几乎相同"的标签
- 同一概念被 AI 标注为 3+ 种不同标签变体
- 用户（自己）浏览时看到标签却不确定它跟另一个有什么区别

**预防策略**：
- 建立 **标签规范表（Controlled Vocabulary）**，每个标签有唯一 canonical form + aliases 列表
- 设计 tag schema 时区分**维度**（dimension）：上墨方式、笔尖材质、产地、价位、品牌……每个维度独立管理，避免扁平化
- AI 标注时强制从已有标签库中选取（closed-set extraction），仅在置信度低时建议新标签
- 定期运行去重/合并脚本（基于 embedding 相似度或编辑距离）

**对应阶段**：Phase 1（schema 设计）— 但需要 Phase 2 做持续治理

## 3. 过度建模（Over-Modeling）

**问题**：知识图谱项目的经典死法——花数月设计完美本体论（ontology），类型层次、属性约束、推理规则一应俱全，然后发现真实数据不整齐、维护成本超预期、项目死在"还没完善 schema"的阶段。

**警示信号**：
- Schema 讨论超过 3 天还在加新字段
- 节点类型 > 10 种且还在增加
- 为了"未来扩展性"添加当前用不上的关系类型
- 觉得需要 OWL/RDF 那套完整语义 Web 堆栈

**预防策略**：
- **最小可用 schema**：v1 只需要 3 种核心节点（Entity / Tag / Source）+ 2 种关系（has_tag / links_to）
- 属性用 JSON blob 存，不做强约束——数据进来再说
- Schema 演进靠 migration，不靠一步到位
- 用 property graph（如 SQLite + JSON）而非 RDF triplestore——个人项目用不上推理引擎

**对应阶段**：Phase 1（架构设计）— 有意识地限制 scope

## 4. 图可视化的"毛线球"问题（Graph Viz Hairball）

**问题**：PROJECT.md 已识别"全局图变毛线球"风险，但局部图也会踩坑——节点超 20 个布局就混乱，交互卡顿，且可视化往往 **看起来好看但没有信息价值**。

**警示信号**：
- 花大量时间调力导向布局参数（spring constants）
- 用户（自己）看着图却不知道下一步该点哪里
- 渲染 > 50 节点时浏览器卡顿
- 可视化成了 demo 而非日常使用的工具

**预防策略**：
- **局部图严格限制展示层级**：只显示 1-hop 邻居，最多 2-hop（需用户手动展开）
- 节点按维度着色/分组，不做通用力导向——改用分层布局或径向布局
- **优先做列表/卡片式漫游**（类似 Wikipedia sidebar），图只是辅助
- 第一版用简单的 adjacency list 渲染（如 D3 force 或 Cytoscape.js），不碰 3D/WebGL
- 设硬指标：单次渲染节点 ≤ 30，超过时自动折叠为类别节点

**对应阶段**：Phase 2-3（前端漫游体验）

## 5. AI 标注质量黑箱（AI Tagging Quality Trap）

**问题**：LLM 标注看起来"差不多对"，但错误会累积——错误标签导致错误链接，错误链接导致漫游体验混乱。尤其钢笔领域有大量专业术语和微妙区分（如「弹性」vs「软弹」vs「笔尖柔软」），通用模型容易混淆。

**警示信号**：
- 批量标注后未做抽样验证
- 标注 prompt 没有领域特定的 few-shot examples
- AI 经常生成不在 controlled vocabulary 中的新标签
- 浏览时发现明显错标（金属笔杆被标为"树脂材质"）

**预防策略**：
- **黄金标准集**：手动标注 50+ 条作为 evaluation benchmark，每次改 prompt 必须跑 benchmark
- 标注结果分三档：高置信度（自动入库）/ 中置信度（待审）/ 低置信度（丢弃）
- 输出格式强制 structured output（JSON schema），只允许从已有标签列表中选取
- 对每个标注结果记录 provenance（来源 + 时间戳 + 模型版本 + prompt hash），方便回溯修正
- 在 pipeline 中加入 **冲突检测**：如果同一实体的两次标注结果矛盾，标记为需要人工审核

**对应阶段**：Phase 1（导入 pipeline）+ 持续运营

## 6. 远程 Agent 执行风险（Remote Agent Miscommunication）

**问题**：规划本机、执行远程——这个开发模式下，Agent 对领域知识理解不足时会做出"合理但错误"的决策。数据建模错误尤其危险：比如把"上墨方式"建模为 boolean 而非 enum，或者把品牌和系列建模为同一节点类型。

**警示信号**：
- Agent 实现与预期偏差，但代码"能跑"所以没被发现
- 数据库里出现脏数据（如日期格式不一致、枚举值大小写混乱）
- Agent 自行决定了某个设计选择但没有报告

**预防策略**：
- **每个 Task 附带验收测试**（assertion-level），不只是"跑起来"而是"数据正确"
- 关键 schema 决策写入 `DECISIONS.md`，Agent 必须在执行前确认理解
- 第一批数据导入后，人工抽查 10% 确认质量
- Task 粒度控制：单个 task 不超过 2-3 小时工作量，交付后立即 review

**对应阶段**：全程

## 7. 链接密度不足（Sparse Linking）

**问题**：知识图谱的核心价值是链接，不是孤立词条。如果内容录入时只做了基本标签但没建立词条间的交叉链接，结果就是"带标签的数据库"而非"可漫游的图谱"。

**警示信号**：
- 实体页面平均出链 < 3
- 大量词条只能通过搜索/筛选到达，无法从其他词条跳转过来
- 品牌页面没有链接到旗下笔款；笔款页面没有链接到同类竞品

**预防策略**：
- 除了 `has_tag` 关系，显式建立 **实体间链接**：`made_by`（品牌）、`similar_to`（竞品）、`successor_of`（系列演进）、`mentioned_in`（出处）
- AI 标注 pipeline 中增加 **relation extraction** 步骤——不只提取标签，也提取实体间关系
- 设置内容质量指标：词条发布前检查 outgoing links ≥ 3
- 在前端实现 "相关词条" 自动推荐（基于共享标签的 Jaccard 相似度），补充手动链接的不足

**对应阶段**：Phase 1（schema 中预留关系类型）+ Phase 2（推荐算法）

## 8. 导入 Pipeline 脆弱性（Brittle Ingestion Pipeline）

**问题**：内容来源多样（Markdown、HTML 爬取、PDF 书籍、手动笔记），每种来源格式不同。如果为每种来源写定制化 parser 且没有统一中间格式，pipeline 极易崩溃。

**警示信号**：
- 每新增一个来源就要改核心代码
- 同一内容经过不同路径导入后结果不同
- Parser 对格式变化敏感（网站改版即挂）

**预防策略**：
- 定义 **统一中间格式**（canonical document format）——所有来源先转为同一结构（如 `{title, body_markdown, source_url, raw_metadata}`），再进入标注 pipeline
- Source adapter 模式：每个来源一个轻量 adapter（只负责转中间格式），核心 pipeline 只处理中间格式
- 为每个 adapter 写 **snapshot test**（固定输入 → 固定输出），来源格式变化时立即发现
- 幂等导入：相同内容重复导入不产生重复节点（基于 content hash 或 source_url 去重）

**对应阶段**：Phase 1（pipeline 架构）

## 9. 搜索/查询体验与漫游体验的冲突（Search vs. Browse Tension）

**问题**：搜索优化和漫游优化是两种不同的信息架构。搜索要精准命中，漫游要 serendipity（意外发现）。如果只优化搜索（全文检索 + 筛选），漫游体验会很差；反之亦然。

**警示信号**：
- 首页是搜索框而非推荐/热门/随机入口
- 每次使用都依赖"知道要搜什么"
- 没有"随机漫步"或"相关推荐"功能

**预防策略**：
- **漫游入口优先**：首页展示随机精选词条、最近更新、主题聚合，而非空搜索框
- 每个词条页面底部/侧边栏有 "相关词条"（基于标签重叠）和 "随机跳转"
- NL 查询（AI 对话）作为 Phase 2 功能，Phase 1 聚焦浏览路径
- 实现 "breadcrumb trail" — 记录浏览路径，支持回溯

**对应阶段**：Phase 2（前端体验设计）

## 10. 技术选型镀金（Technology Gold-Plating）

**问题**：个人项目用上 Neo4j + Elasticsearch + Vector DB + GraphQL + Next.js + Kubernetes……技术栈越多，维护成本越高，远程 Agent 执行复杂度越大。

**警示信号**：
- 本地开发需要 docker-compose 起 5+ 个服务
- "先部署基础设施再开始开发"
- 选型讨论中出现"生产级""企业级"等词汇

**预防策略**：
- **SQLite + FTS5** 能覆盖 90% 需求（存储 + 全文搜索 + JSON 字段）
- 图关系用邻接表模式（`edges` 表）而非专用图数据库
- 向量搜索等 AI 功能可以后加（Phase 2），先不引入向量数据库
- 部署目标：单个进程 + SQLite 文件，可在任何 VPS 上 `python main.py` 跑起来
- 决策依据：远程 Agent 能否在 30 分钟内 from scratch 搭起开发环境？不能就太复杂了

**对应阶段**：Phase 1（技术选型决策）

## 核心原则（Anti-Pitfall Mantras）

1. **Content-first, system-second** — 没有 300 个词条之前，不做前端美化
2. **Closed vocabulary, open content** — 标签有限可控，内容无限扩展
3. **SQLite-until-proven-otherwise** — 遇到实际瓶颈再升级，不预设
4. **Every link is a browsing path** — 每条关系都应该对应一个用户可能的跳转
5. **AI assists, human verifies** — 自动标注 ≠ 自动信任
